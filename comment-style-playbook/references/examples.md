# Examples

## Good

- `说明该服务负责订单状态流转与外部回调协调，不直接处理数据库写入细节`
- `说明该字段单位为分，避免与元混用`
- `说明该异步任务失败后会重试三次，并在最终失败时写入告警日志`
- `说明该缓存删除必须发生在数据库提交成功之后`

## Bad

- `// save order`
- `// render list`
- `// set loading true`
- `// call api`

## Rule Of Thumb

如果注释删掉之后，读代码的人仍然完全知道“为什么这样写”，那它大概率不该存在。  
如果删掉之后，别人会误解约束、忽略副作用、改坏业务边界，那这条注释就有价值。
