# Admin Web Layout

## Layout Baseline

适用于中大型后台、运营平台、审核系统、数据管理系统。

- 左侧 Logo 区高度默认 `68px`
- 顶部主导航高度默认 `68px`
- 顶部导航和标签栏固定，不跟随内容区滚动
- 左侧菜单和右侧内容区各自独立滚动
- 页面整体优先采用中后台常见的左右分栏结构

## Navigation

- 复杂项目优先使用分栏布局或多级菜单
- 一级导航负责业务域切换，不要把全部菜单平铺到一个层级
- 左侧菜单负责当前业务域内导航
- 当前页路径要清楚，避免用户迷路
- 图标只做辅助识别，不取代文字标签

## Tag Bar

- 顶部导航下方保留访问过页面的标签栏
- 标签栏单独占一整行
- 最右侧放快捷功能图标
- 点击图标后弹出气泡菜单，支持关闭当前、关闭其他、关闭左侧、关闭右侧、关闭全部
- 标签文案要可读，不要只留图标

## Content Area

- 内容区通常分为筛选区、列表区、详情区或操作区
- 筛选区放在内容顶部，结构清晰，常用筛选条件优先露出
- 列表过宽时，操作列固定，关键识别列尽量靠左
- 非关键宽列允许横向滚动
- 分页、批量操作、导出等次级操作与表格关系要清楚
- 详情查看优先考虑抽屉或页内分区，避免动不动整页跳转打断任务流
- 高频操作优先靠近列表或详情上下文，不要让用户来回找按钮

## Table Rules

- 表格优先服务效率，不要为了“好看”牺牲扫描速度
- 列标题短而明确
- 状态、金额、时间、负责人等关键字段要形成稳定扫视路径
- 操作列固定在右侧
- 重要状态优先用标签或轻量强调，而不是堆整段说明文案
- 批量选择、批量操作和单条操作要明显区分
- 危险操作要二次确认，但不要让所有操作都变成弹窗地狱

## Forms And Detail

- 表单默认按任务分组，不要把所有字段堆成一整页
- 长表单优先分区、分步或折叠，而不是无限向下滚
- 必填、选填、默认值和格式要求要清楚
- 查看详情与编辑详情的层级要明确，避免在阅读态和编辑态之间来回混乱切换
- 需要频繁对照列表的编辑任务，可优先使用抽屉而不是整页跳转

## Actions And Feedback

- 一个区域通常只保留一个主按钮
- 次要按钮、文本操作、危险操作层级要拉开
- 成功、失败、处理中、无权限、无数据都要有明确反馈
- 空状态要告诉用户下一步能做什么，但不要写成说明书
- 加载态优先用骨架屏、局部 loading 或表格 loading，不要整页空白闪烁

## Sticky And Scroll

- 顶部主导航固定
- 标签栏固定
- 左侧菜单独立滚动
- 右侧内容区独立滚动
- 表格横向滚动不影响顶部导航和标签栏位置

## Page Density

- 后台不怕信息密度，但怕没有秩序
- 优先用对齐、分组、留白和层级解决复杂度
- 不要把后台页面做成只有大卡片和大留白的展示页
- 桌面端后台应优先保证主要工作区宽度，不为移动端展示强行牺牲桌面效率
