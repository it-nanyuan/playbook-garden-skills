# Discovery

## Goal

在设计开始前，用最少的问题补齐最关键的信息，避免一上来就画出空洞的大标题卡片页。

## Ask First

优先确认下面 5 类信息：

1. 这是后台网页、控制台、数据看板，还是移动 App
2. 核心用户是谁，他们最常执行什么任务
3. 有没有竞品、参考产品或现成系统截图
4. 需要偏稳重企业、轻量现代，还是高密度数据后台
5. 是新项目还是已有项目，当前技术栈和组件库是什么

## If The User Cannot Describe Style

用更容易回答的问题代替抽象审美提问：

- 你觉得哪些产品看起来比较专业
- 更接近阿里云控制台、飞书后台、Stripe Dashboard，还是 Notion 风格
- 更看重信息密度还是更看重轻松感
- 更偏业务执行效率，还是更偏展示汇报

## Ask Narrowly

默认只问 3 到 5 个关键问题。  
如果已经能从仓库、截图或上下文推断答案，不要重复追问。

## Output After Discovery

在拿到答案后，先形成这些判断，再继续设计：

- 页面属于哪类业务系统
- 最关键的操作流和主任务
- 信息密度级别
- 适合的布局模式
- 技术栈和组件库方向
