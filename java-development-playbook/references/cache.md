# Cache

## Goal

缓存的目标是降低热点访问成本和提升吞吐，不是为了“显得架构高级”。

## Rule

- 先确认是否真的需要缓存
- 统一 key 命名规范
- 统一 TTL 策略
- 明确缓存更新、删除、预热和失效逻辑
- 缓存异常不能悄悄破坏主流程语义

## Good Bias

- 热点查询、配置项、字典数据、短期聚合结果适合缓存
- 涉及一致性时先明确策略，再写代码
- 防止缓存穿透、击穿、雪崩要有基本意识

## Key Naming

- Redis key 前缀建议至少包含：系统名、业务域、资源类型
- 示例：`app:order:detail:{id}`
- 多环境场景需要明确环境隔离时，可额外加入环境前缀
- 同一业务不要同时出现多套 key 风格

## Avoid

- 随手 set 缓存但没人知道什么时候删
- 一个业务多套 key 规则
- 缓存失败直接吞掉影响又没人知道
